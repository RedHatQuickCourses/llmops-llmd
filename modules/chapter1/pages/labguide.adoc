== Deploy using Distributed Inference with llm-d

We will deploy a IBM Granite4 micro model using the `LLMInferenceService` CR. This resource tells the operator to spin up the vLLM pods and the Inference Scheduler.

.Step 1: Create the Project
[source,bash]
----
oc new-project my-ai-lab
----

.Step 2: Apply the LLMInferenceService
Save the following as `llm-inference-service.yaml` and apply it.

[source,yaml]
----
apiVersion: serving.kserve.io/v1alpha1
kind: LLMInferenceService
metadata:
  name: granite4
  namespace: my-api-lab
spec:
  replicas: 1
  model:
    uri: hf://ibm-granite/granite-4.0-micro
    name: granite-4.0-micro
  router:
    scheduler:
      template:
        tolerations:
          - key: "nvidia.com/gpu"
            operator: "Exists"
            effect: "NoSchedule"
        containers:
          - name: main
            args:
              - -v=4 # Debug level
              - --poolName
              - "{{ ChildName .ObjectMeta.Name `-inference-pool` }}"
              - --poolNamespace
              - "{{ .ObjectMeta.Namespace }}"
              - --zap-encoder
              - json
              - --grpcPort
              - "9002"
              - --grpcHealthPort
              - "9003"
              - --secureServing
              - --modelServerMetricsScheme
              - https
              - --modelServerMetricsHttpsInsecureSkipVerify
              - --certPath
              - /etc/ssl/certs
              - --configText
              - |2
              
                apiVersion: inference.networking.x-k8s.io/v1alpha1
                kind: EndpointPickerConfig
                plugins:
                - type: single-profile-handler
                - type: max-score-picker
                - type: queue-scorer
                - type: kv-cache-scorer
                - type: prefix-cache-scorer
                  parameters:
                    hashBlockSize: 64             # block size 16 * 4 chars / token
                    maxPrefixBlocksToMatch: 256   # this matches up to ~4k tokens
                    lruCapacityPerServer: 39744   # total blocks (635904) / block size (16)
                schedulingProfiles:
                - name: default
                  plugins:
                    - pluginRef: prefix-cache-scorer
                      weight: 5.0
                    - pluginRef: queue-scorer
                      weight: 1.0
                    - pluginRef: kv-cache-scorer
                      weight: 1.0
                    - pluginRef: max-score-picker
    route: { }
    gateway: { }
  template:
    tolerations:
      - key: "nvidia.com/gpu"
        operator: "Equal"
        value: "True"
        effect: "NoSchedule"
    containers:
      - name: main
        env:
          - name: VLLM_ADDITIONAL_ARGS
            value: "--disable-uvicorn-access-log"
        resources:
          limits:
            cpu: '1'
            memory: 8Gi
            nvidia.com/gpu: 1
          requests:
            cpu: '1'
            memory: 8Gi
            nvidia.com/gpu: 1
        livenessProbe:
          httpGet:
            path: /health
            port: 8000
            scheme: HTTPS
          initialDelaySeconds: 120
          periodSeconds: 30
          timeoutSeconds: 30
          failureThreshold: 5
----

[source,bash]
----
oc apply -f manifests/llm-inference-service.yaml
----

== Phase 2: Expose and Secure (The Gateway)

In OpenShift AI 3.0, you must explicitly configure the route if the UI didn't auto-associate it.

.Step 1: Verify the HTTPRoute
[source,bash]
----
oc get httproute -n my-ai-lab
----
*Ensure the route points to the `openshift-ai-inference` gateway.*

.Step 2: Get the Authentication Token
Since `llm-d` is secured by default, retrieve the ServiceAccount token created for this model.
[source,bash]
----
# The secret name usually follows the pattern <model-name>-sa-token
SECRET_NAME=$(oc get serviceaccount qwen-7b-distributed-sa -o jsonpath='{.secrets[0].name}')
export TOKEN=$(oc get secret $SECRET_NAME -o jsonpath='{.data.token}' | base64 -d)
----

.Step 3: Get the Inference URL
[source,bash]
----
export INFERENCE_URL=$(oc get httproute qwen-7b-distributed -o jsonpath='{.spec.hostnames[0]}')
echo "Targeting: https://$INFERENCE_URL"
----

== Phase 3: Verify Intelligent Routing (The Test)

We will prove that the scheduler is routing based on cache affinity.

.Step 1: The "Cold" Request (Baseline)
Send a request with a unique system prompt. Measure the total time.
[source,bash]
----
time curl -k -X POST "https://$INFERENCE_URL/v1/chat/completions" \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "qwen-7b",
    "messages": [
      {"role": "system", "content": "You are a helpful assistant talking about OpenShift AI."},
      {"role": "user", "content": "Explain Kubernetes in 50 words."}
    ]
  }'
----

.Step 2: The "Warm" Request (Cache Hit)
Send the **exact same** request immediately. The Scheduler should recognize the prefix hash and route to the same pod.
[source,bash]
----
# Run the exact same command again.
# Compare the 'real' time. The second request should be significantly faster (lower TTFT).
----

== Phase 4: Observability (Golden Signals)

Validate your test using the Dashboard metrics.

1.  Navigate to **OpenShift AI Dashboard** -> **Deployments**.
2.  Select `qwen-7b-distributed`.
3.  Click the **Endpoint Performance** tab.
4.  Set **Refresh Interval** to `15 seconds`.
5.  Look for:
    * **Request Count:** You should see your 2 spikes.
    * **Avg Response Time:** The second spike should be lower than the first.
